# 正常系统端实验性窗口背景

此功能只影响正常 Windows 系统端，不会写入或改变 PE 端界面。默认关闭，可在“关于”页勾选“启用 Mica（实验性）”；程序会同步保存到同目录的 `config.json`：

```json
{
  "experimental_window_backdrop": "mica"
}
```

配置只保留两个值：

- `none`：关闭全客户区材质并使用原有不透明客户区（默认）；默认 Win32 标题栏仍交给 DWM 自动决定系统背景。
- `mica`：长驻主窗口背景；Windows 11 当前对应 Mica。

`auto`、`acrylic`、`mica_alt` 是早期实验值，现已停止支持；读取这些历史值或任何未知值都会安全回退 `none`，不会导致整个配置文件失效。关闭实验选项时程序写入公开的 `DWMSBT_AUTO` 并撤销客户区 frame：DWM 可只在默认标题栏后应用系统背景，也可按系统策略不应用。只有用户明确启用 Mica 时才会继续检查 `DwmIsCompositionEnabled`，写入 `DWMSBT_MAINWINDOW`，通过 `DwmGetWindowAttribute` 回读同一公开属性，并用 `DwmExtendFrameIntoClientArea` 覆盖完整窗口。配置值只是请求，不能作为“材质实际存在”的证明。该系统背景枚举最低支持 Windows 11 build 22621；旧系统、关闭 DWM 合成、节能/辅助功能策略、属性回读不一致或 DWM 拒绝请求时，程序会记录警告，恢复 `DWMSBT_AUTO`、零 frame 与普通不透明控件主题。详见微软的 [DWM_SYSTEMBACKDROP_TYPE](https://learn.microsoft.com/windows/win32/api/dwmapi/ne-dwmapi-dwm_systembackdrop_type)、[DWMWINDOWATTRIBUTE](https://learn.microsoft.com/windows/win32/api/dwmapi/ne-dwmapi-dwmwindowattribute) 与 [DwmSetWindowAttribute](https://learn.microsoft.com/windows/win32/api/dwmapi/nf-dwmapi-dwmsetwindowattribute)。

Mica 的激活与失活必须和顶层 DWM 背景同步。微软规定 `WM_NCACTIVATE` 的 `wParam=TRUE/FALSE` 分别表示标题栏应显示为激活/失活状态；程序以这条消息维护每个主窗口或工具对话框自己的激活态，并在同一次冻结重绘事务中切换材质/普通控件调色板。它不会在消息过渡期调用可能仍指向旧窗口的 `GetForegroundWindow` 再猜一次状态。材质控件表面只有在顶层 `DWMSBT_MAINWINDOW` 回读确认、全客户区 frame 成功且该顶层窗口当前激活时才启用；失活后按钮、字段和列表立即改回普通主题，避免“标题栏已失去 Mica、控件仍独立悬浮”的混合状态。正常端主窗口和工具对话框请求全窗口 Mica 前还会显式排除 WinPE；PE 端本身不引用 backdrop 实现，始终保持普通不透明原生主题。参见微软的 [WM_NCACTIVATE](https://learn.microsoft.com/windows/win32/winmsg/wm-ncactivate) 与 [Win32 Mica 指南](https://learn.microsoft.com/windows/apps/desktop/modernize/ui/apply-mica-win32)。

明暗主题启用 `mica` 后，主窗口、导航基础层和正常端工具对话框都使用 DWM 的黑色玻璃键，材质覆盖完整客户区。普通导航和次要按钮直接使用蓝灰预乘 Alpha 表面；深色常态表面提高覆盖率，避免壁纸把控件冲得过透，浅色常态表面使用非纯白的冷灰低覆盖层，避免退化成纯白矩形。悬停、按下和禁用态使用独立但克制的叠加色与透明度，选中导航和主操作按钮仍保持不透明强调色。Edit、闭合 ComboBox、ListBox、ListView 和表头共享同一材质状态定义；经典 Win32 子 HWND 无法可靠地把逐像素 Alpha 发布到父 DWM 表面，因此使用该覆盖层在 Windows 11 Mica 中性基底上的确定性合成色。按钮保留原有 4 倍 GDI `RoundRect`，字段和列表外框使用固定 8×8 子像素覆盖率绘制同半径圆角；原生输入、光标、键盘和无障碍语义保持不变。

浅色全客户区不是通过把黑色改成近黑色实现的。经典 STATIC 子 HWND 在全客户区玻璃与 DWM 重定向组合下不能稳定保留透明文字贡献，因此程序把每个浅色材质文本标签的 Windows 11 中性材质回退色、当前字体和当前文字色画入同一兼容位图，再一次 `BitBlt` 发布；这避免文字区域先清空、后补字造成闪烁，同时视觉底色与 DWM 失活时的中性基底一致。实体按钮、复选/单选标题和闭合 ComboBox 使用白色 GDI 字形遮罩重着色为预乘 BGRA，其中 ClearType RGB 遮罩按三通道平均覆盖率转换，保留普通 400 字重而不把导航与按钮伪画成粗体；浅色材质复选框继续逐像素使用固定 Windows 11 主题资源的原始图元，只把透明和半透明边缘与 Windows 11 中性材质回退色预混合，并把真正不透明的纯黑笔画与 DWM 黑色玻璃键区分开，不改变半径、勾号、状态或配色；ListView 自绘行则在已知实体背景上使用不透明 GDI 文字，保留与原生列表一致的 ClearType 子像素渲染。Edit、ComboBox、ListBox 和 ListView 不再使用圆角 HRGN；`CreateRoundRectRgn` 和 `FrameRgn` 的二值边缘会裁掉部分覆盖率像素，在直边与圆弧连接处产生阶梯、颗粒或 C 形断口。库存客户区绘制完成后，统一的 8×8 覆盖率外框最后写入：全外角恢复 DWM 黑色玻璃键，部分覆盖像素通过 32 位顶向下 BGRA DIB 发布预乘 Alpha，直边和圆弧使用同一个绝对边框色；ComboBox 的矩形窗口区域只隐藏 USER32 保留的展开高度，并在裁剪后显式重绘刚暴露的父窗口尾部，不能让旧 UxTheme 像素残留成字段下横线。浅色材质使用可辨识的一像素描边，空 ListView 显式填满完整客户区。每次 GDI 将字形画入位图后都先调用 `GdiFlush`，再读取或改写像素，避免延迟 GDI 批处理在真实多行列表首帧中破坏 USER32 绘制状态。微软对 Mica 作为控件后方基础层、系统背景、整张玻璃、GDI 区域和 GDI 批处理同步的说明可参考：[Mica material](https://learn.microsoft.com/windows/apps/design/style/mica)、[DWM_SYSTEMBACKDROP_TYPE](https://learn.microsoft.com/windows/win32/api/dwmapi/ne-dwmapi-dwm_systembackdrop_type)、[DwmExtendFrameIntoClientArea](https://learn.microsoft.com/windows/win32/api/dwmapi/nf-dwmapi-dwmextendframeintoclientarea) 与 [CreateDIBSection](https://learn.microsoft.com/windows/win32/api/wingdi/nf-wingdi-createdibsection)。PE 端始终使用普通不透明背景。

扩展玻璃客户区中的原生单行 Edit 不使用 `WS_EX_LAYERED` 或 `SetLayeredWindowAttributes`。微软说明分层窗口由系统保存离屏图像，并不保证窗口内容重新暴露时再次请求绘制；这类子窗口因此不能满足“首次显示和主题切换后不依赖鼠标消息”的要求。程序在每次主题应用时主动清除旧版本可能遗留的分层样式，保留 USER32 原生文字、光标、选择、IME、键盘、命中和无障碍语义，并让 Edit 与 ComboBox、ListView 一起进入窗口的同步首帧及主题重绘事务。Edit 的 `WM_PAINT` 先在兼容位图中填入确定性字段色，再调用原生 `WM_PRINTCLIENT` 绘制文字、光标和选择，`GdiFlush` 后一次 `BitBlt` 到窗口 DC；创建兼容位图失败时才直接调用原生绘制。该子控件不安装圆角 HRGN，外角恢复父窗口玻璃键，再由统一 8×8 覆盖率路径绘制圆角外框。ComboBox、ListBox 和 ListView 的不透明 BGRA/主题颜色路径不使用 Edit 画刷补偿。闭合 ComboBox 的正文、箭头区和圆角外框始终从同一次状态查询取得 normal/hot/dropped 表面色，禁止外框用常态色而正文用展开色，否则抗锯齿内沿会产生白色弧线。悬停、聚焦和展开状态还必须把圆角外框同步切换为当前明暗调色板的强调色；Windows 10/11 的库存 ComboBox 在样式边框移除后仍可能保留主题客户区内缩，因此同步闭合重绘必须通过 window DC 覆盖整个可见字段后再画唯一外框。真实鼠标点击会进入 USER32 的嵌套下拉跟踪循环，库存主题可能在按下消息返回前再次覆盖闭合框；实现必须把同步重画投递到该嵌套循环中，并用真实鼠标点击截图回归，不能只用 `CB_SHOWDROPDOWN` 验证。

系统背景不是窗口透明度。主窗口控件树先在隐藏状态完成字体、主题和布局准备；首次显示时允许临时添加 `WS_EX_LAYERED`，但全局 alpha 必须是非零的 1，确保顶层窗口从第一帧起仍拥有命中测试，不会把导航点击穿透给窗口后方。程序在该近透明屏障后同步重绘完整非客户区、客户区与全部子控件，再恢复 alpha=255、立即移除 `WS_EX_LAYERED` 并用 `SWP_FRAMECHANGED` 刷新命中与非客户区缓存；精简 WinPE 拒绝分层属性时退回同一同步重绘。alpha=0 和 `WS_EX_TRANSPARENT` 都禁止使用。页面、语言和主题切换不得再次进入分层状态，而是先记录当前可见后代，分别暂停这些子 HWND 与顶层根窗口，在冻结期完成文字、库存、显隐和布局，再恢复仍应可见的后代并以一次 `RedrawWindow(...RDW_ALLCHILDREN | RDW_UPDATENOW)` 发布完整树。主窗口明确不使用 `WS_EX_COMPOSITED`：该标志的自底向上整树合成会把 Edit 光标等局部更新扩大成可观察的逐控件中间帧。页面切换只更新当前页和公共页头，不再重复布局所有隐藏页面；常用材质圆角图元由有界线程本地缓存复用，避免切页时重复构造相同位图。微软关于 Mica 基础层、分层窗口命中与批量重绘的说明见 [Apply Mica in Win32 apps](https://learn.microsoft.com/windows/apps/desktop/modernize/ui/apply-mica-win32)、[Extended Window Styles](https://learn.microsoft.com/windows/win32/winmsg/extended-window-styles)、[Layered Windows](https://learn.microsoft.com/windows/win32/winmsg/window-features)、[WM_SETREDRAW](https://learn.microsoft.com/windows/win32/gdi/wm-setredraw) 与 [RedrawWindow](https://learn.microsoft.com/windows/win32/api/winuser/nf-winuser-redrawwindow)。

## ListView 刷新、选择与滚动条边界

所有原生 ListView 保留 `LVS_EX_DOUBLEBUFFER`；微软明确说明它只减少单次绘制闪烁，并不把一串逐项插入自动合并成业务事务。硬件信息后台结果在同一窗口消息中完成删除和整批插入，期间不返回消息循环，最后只用一次带 `RDW_ERASE | RDW_FRAME | RDW_INVALIDATE | RDW_ALLCHILDREN | RDW_UPDATENOW` 的 `RedrawWindow` 同步发布。这里刻意不向硬件页的子 ListView 单独发送 `WM_SETREDRAW(FALSE/TRUE)`：微软文档说明 `DefWindowProc` 会通过移除/恢复 `WS_VISIBLE` 实现该消息；在 DWM 重定向的 report/header 父子 HWND 上，这会允许一个暂时没有 Header 的可见合成帧。页面切换则属于整棵树事务：在根窗口仍冻结时恢复先前记录的可见后代，最后由根级一次 `RedrawWindow` 一起发布。参见 [Extended List-View Styles](https://learn.microsoft.com/windows/win32/controls/extended-list-view-styles) 与 [WM_SETREDRAW](https://learn.microsoft.com/windows/win32/gdi/wm-setredraw)。

安装分区选择只处理 `LVN_ITEMCHANGED` 所携带 `NMLISTVIEW.uChanged` 中的 `LVIF_STATE`，并比较 `uOldState/uNewState` 的 `LVIS_SELECTED` 位。一次选择移动通常先通知旧行取消、再通知新行选中；程序只向主消息队列投递一个合并任务，在 ListView 自己的同步通知/绘制事务结束后读取最终选择，并在一轮根窗口重绘事务中更新依赖状态和布局。文本、焦点或未改变选中位的通知不再触发页面刷新。参见 [LVN_ITEMCHANGED](https://learn.microsoft.com/windows/win32/controls/lvn-itemchanged)。

Report Header 是 ListView 的真实子 HWND。ListView 统一增加 `WS_CLIPCHILDREN`，避免父窗口 body 在 Header 已完成后再次覆盖子窗口区域；Header 的 UxTheme 动画绘制被关闭，由唯一自绘入口负责，ListView 默认绘制结束后只同步恢复 Header 和确定性圆角外框，Header 绘制绝不反向失效整张列表，因此不会形成定时重绘反馈环。该裁剪关系遵循微软的 [Window Features](https://learn.microsoft.com/windows/win32/winmsg/window-features) 说明。

滚动、键盘、命中和无障碍仍由 comctl32 v6 原生 ListView/滚动条处理。`WM_HSCROLL`、`WM_VSCROLL`、滚轮和 UxTheme 滚动条动画定时器等高频消息直接交给默认过程，不能在每个 `SB_THUMBTRACK` 中失效整表、同步补画 Header 或直接覆盖重定向表面；正常 `WM_PAINT/WM_NCPAINT` 再恢复唯一外框。深色报表选择公开的 `DarkMode_Explorer` 主题族，浅色使用 `Explorer`；不接管滚动条几何或输入，也不使用 FlatSB 颜色覆写。微软文档说明 `FlatSB_SetScrollProp` 只支持 comctl32 4.71–5.82，而本程序清单启用 v6，因此这种旧接口不能作为当前滚动条着色方案。参见 [WM_VSCROLL](https://learn.microsoft.com/windows/win32/controls/wm-vscroll)、[Extended List-View Styles](https://learn.microsoft.com/windows/win32/controls/extended-list-view-styles) 与 [FlatSB_SetScrollProp](https://learn.microsoft.com/windows/win32/api/commctrl/nf-commctrl-flatsb_setscrollprop)。当前用户实机仍报告拖动滚动条时闪烁，因此 `TODO.md` 保持该项未完成；本段只记录已采用的公开消息边界，不把它写成已通过的视觉结论。
